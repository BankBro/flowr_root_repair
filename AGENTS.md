# AGENTS.md

## 项目定位

- 本仓库以 FLOWR.ROOT 为基线, 研究蛋白质-配体 clash 的诊断与局部坐标修复.
- FLOWR.ROOT 原有训练, 生成和预测行为是需要保留的基线. 修复功能必须显式启用, 不得静默改变默认行为.
- `docs/STATUS.md` 是当前项目状态入口. 开始任务前先阅读它, 再阅读其中指向的当前计划和距离目标文件最近的 `AGENTS.md`.
- `AGENTS.md` 只维护稳定的协作和工程规则. 模型设计, 训练方案和实验结论写入 `docs/plan/`, `docs/report/` 与实验记录, 不在规则文件中重复维护.

## 协作与文本

- 与用户及仓库内协作型 Agent 始终使用中文交流.
- 所有写入仓库或终端的文本使用 UTF-8 编码. 中文说明使用英文标点.
- 能从代码, 配置或文件系统确认的事实应先检查, 不凭印象回答.
- 仓库可能包含用户尚未提交的修改. 不得覆盖, 回退或删除与当前任务无关的改动.

## 代码边界

- 新增修复实现优先放入 `flowr/repair/`. 只有确有必要时才修改 FLOWR.ROOT 共享模块.
- 修改共享模块时保持改动范围最小, 明确区分原有行为和新增修复行为, 并验证原有路径没有发生回归.
- 测试范围按改动风险决定. 正式训练或长时间实验前, 必须先完成轻量 CPU 或 GPU 检查以及必要的 smoke test.
- 根目录 `scripts/` 保留跨实验复用脚本. 稳定的修复脚本未来放入 `scripts/repair/`, 单次实验专用脚本放入对应实验目录.

## 文档与实验

- 计划文件使用 `docs/plan/YYYYMMDD-NN-<slug>-plan.md`.
- 报告文件使用 `docs/report/YYYYMMDD-NN-<slug>-report.md`.
- `NN` 固定为两位数, 如 `01`, `02`. plan 和 report 使用各自创建时的日期与编号, 通过相同 slug, 明确链接和 `experiment_id` 建立对应关系.
- plan, experiment, report 严格一一对应. 一个总实验可以包含多个预先规划的子阶段, 但失败或中止的总实验仍需要报告.
- 计划开始指导实现后即作为历史基线保留, 不静默改写原始目标. 一般偏差记录在报告中, 路线发生重大变化时新建计划.
- `docs/STATUS.md` 只保留当前项目快照. `docs/EXPERIMENT_LOG.md` 按时间追加实验和阶段记录, 不重写历史记录.
- 实验目录使用 `experiments/<experiment_id>/`, 对应输出使用 `outputs/<experiment_id>/`. 具体规则见各目录的 README 和 `AGENTS.md`.
- 不强制为每个实验建立固定 metadata, schema 或 `configs/` 结构. 只有真实需要时才增加, 目录组织以简单, 清楚, 可追溯为准.

## 临时产物

- 纯调试用的临时脚本和文件放入 `tmp/YYYYMMDD-<task-slug>/`.
- 临时任务结束后删除对应子目录. 会影响研究决策的 smoke test 不应只留在 `tmp/`, 而应在正式实验建立后纳入实验记录.
